Capítulo 01

Apresentação e Identificação do Software

O SIGE (Sistema de Gestão de Estágios) é uma plataforma integrada de gestão operacional, administrativa e financeira desenvolvida em arquitetura web moderna. O sistema foi projetado para atuar como um Hub corporativo que conecta unidades concedentes (empresas), instituições de ensino (escolas), estagiários, supervisores de estágio, candidatos a processos seletivos públicos (concursos) e operadores administrativos.

O sistema divide-se em três grandes frentes de atuação tecnológica:

  • Gestão de Estágios e Contratos (TCE): Automação completa do fluxo de contratação de estagiários, controle de limites por empresa, aditivos de alteração, rescisões, controle de recesso, avaliações automáticas de desempenho e assinatura digital de documentos via integração com ZapSign.
  • Portal de Vagas e Candidaturas: Painel para empresas publicarem oportunidades de estágio e emprego, com área pública de busca, envio de currículos e sincronização automática com o cadastro de contratos.
  • SIGE Concursos: Módulo completo para elaboração e controle de concursos públicos e processos seletivos. Permite cadastrar órgãos públicos, editais, cargos, locais de prova e salas, executar o algoritmo de alocação automática e balanceada de candidatos por sala, homologar inscrições (com exportação financeira de taxas e isenções) e emitir folhas de presença de provas.
Capítulo 02

Especificação de Arquitetura Tecnológica

A plataforma foi construída seguindo padrões de engenharia de software de alta performance, portabilidade e segurança:

  • Backend Framework: Laravel (PHP 8.1+), empregando o padrão arquitetural MVC (Model-View-Controller), ORM Eloquent para abstração e segurança das transações do banco de dados, e Jobs assíncronos/agendados para tarefas em background.
  • Database Engine: MySQL (5.7 ou superior), utilizando índices otimizados nas chaves de relacionamento, transações SQL acopladas (para garantir a integridade dos dados em caso de falhas) e restrições de chave estrangeira (FK) em nível de banco de dados.
  • Frontend e UI: Blade templates integrados a bibliotecas JavaScript assíncronas (AJAX via Axios/Fetch) e CSS Vanilla estruturado de alto desempenho. A navegação de buscas utiliza Select2 dinâmico para otimizar a renderização de grandes volumes de registros.
  • Portabilidade Móvel (PWA): O sistema opera também como um PWA (Progressive Web App). Possui Service Worker configurado para controle de cache, manipulação offline de ativos essenciais e manifest de instalação que permite que o sistema seja adicionado como aplicativo nativo em dispositivos móveis Android e iOS.

Capítulo 05

Módulo de Estagiários, Escolas e Supervisores

Upload Seguro de Documentos de Identidade

No portal "Meu Perfil", o estagiário anexa cópias digitais de documentos pessoais (Documento de Identidade, Comprovante de Residência e Comprovante Escolar). Para evitar instabilidades e perda de consistência, o fluxo de desenvolvimento implementa:

  • Limite de tamanho aumentado para 5MB (5120 KB) por documento, permitindo arquivos em PDF gerados por scanner sem estouros de cota.
  • Controle transacional no backend: se o upload físico de qualquer arquivo falhar no disco, a transação no banco sofre rollback completo, deletando arquivos parciais e mantendo a integridade.
  • A exclusão do arquivo antigo no storage do servidor só é efetuada após a gravação com sucesso dos novos dados.

Recuperação de Login Automática

Para estagiários que possuem o termo de estágio cadastrado administrativamente mas nunca criaram suas credenciais do portal, o sistema disponibiliza um fluxo baseado na inserção do CPF. O sistema consulta tb_estagiarios e bifurca conforme o status:

  1. CPF não localizado: O sistema redireciona para a tela de novo cadastro.
  2. Cadastro duplicado: Bloqueia a ação por segurança e orienta o estagiário a contatar o suporte via WhatsApp.
  3. E-mail existente: Informa o e-mail de login cadastrado e solicita recuperação de senha clássica.
  4. Cadastro sem e-mail de usuário: Renderiza na tela um formulário seguro de primeiro acesso para o usuário preencher e-mail e nova senha corporativa.

Pesquisa Rápida de Usuários

Painel de listagem de usuários administrativos contendo filtro com busca de termos assíncronas para estagiários e empresas, limitando a visualização em no máximo 50 itens retornados para preservação de processamento.

Capítulo 06

Gestão de Termos (TCE), Alterações e Rescisões

Conceito de Auditoria de Termos (Campos Fixo)

Como contratos e termos de estágio (TCE) possuem validade jurídica, alterações cadastrais posteriores de empresas ou estagiários (como mudança de razão social, CNPJ ou nome) não podem invalidar ou alterar silenciosamente os termos antigos gerados no passado.

Para resolver isso, a tabela tb_termos armazena os dados correntes em relacionamentos, mas espelha permanentemente os dados vitais em colunas terminadas em _fixo:

  • desc_atividades_fixo: Armazena a redação exata do plano de atividades.
  • nome_orientador_fixo e cargo_orientador_fixo: Salva os dados do supervisor.
  • data_fim_estagio_fixo: O prazo final acordado.
  • valor_bolsa_fixo e auxilio_transporte_fixo: Dados financeiros permanentes.
  • horario_fixo: Jornada do estagiário no ato de emissão.

Aditivos e Rescisões de Contratos

As alterações de contratos em andamento criam registros na tabela tb_alteracao_termo, guardando o histórico das configurações financeiras e cronogramas antigos. As rescisões encerram a vigência, atualizam as concessões e geram cálculos de saldos na tabela tb_concessoes_recesso, garantindo a governança do saldo de dias remanescentes para pagamento rescisório.

Além disso, o sistema controla limites individuais de estágios ativos por empresa e restringe visualizações: empresas e estagiários acessam apenas dados associados aos seus IDs, enquanto operadores e administradores dispõem de visão global.

Capítulo 07

Integração ZapSign para Assinatura Eletrônica

Para eliminar o envio físico de papéis e assinaturas manuais, o SIGE implementa integração direta com a API da ZapSign para assinatura digital de termos.

Fluxo Técnico de Assinaturas

  1. O operador cria o termo (TCE) e clica no botão "Enviar para Assinatura Eletrônica".
  2. O sistema gera o documento em PDF, empacota metadados (nome da empresa, estagiário, testemunhas, e-mails de envio) e realiza chamada REST POST para a ZapSign.
  3. A API retorna um código identificador exclusivo (zapsign_doc_token) que é gravado no banco local, alterando o status do termo para "Aguardando Assinatura".

Sincronização via Webhook

O sistema expõe um endpoint público para escuta dos eventos da ZapSign. Quando todas as partes interessadas assinam o documento digitalmente, a ZapSign realiza chamada POST (Webhook) para o SIGE. O backend:

  • Registra os dados recebidos na tabela de log ZapSignWebhookLog.
  • Altera a coluna zapsign_status na tabela tb_termos para "assinado".
  • Habilita instantaneamente o botão de download do PDF assinado para estagiários, operadores e empresas associadas ao contrato.
Capítulo 08

Portal de Vagas e Candidaturas (Vagas de Emprego)

O sistema possui um portal dedicado para a publicação e captação de currículos de estagiários.

Divulgação e Seleção

  • Numeração automática: Cada vaga criada recebe um número sequencial exclusivo gerado anualmente por empresa no formato ANO-SEQUENCIA (ex: 2026-0001).
  • Busca Pública de Vagas: Candidatos e estudantes buscam vagas ativas por atividades ou localidade diretamente no portal de divulgação.
  • Envio de Currículo: O candidato submete sua candidatura anexando currículo em PDF ou DOC (limite máximo de 5MB por arquivo).
  • Controle de Status: A empresa e o operador triam candidatos, alteram o status curricular e escolhem o estagiário vencedor, que recebe notificação imediata por e-mail configurado.

Vinculação com o Cadastro do Contrato

Quando um estagiário é escolhido em uma vaga, o sistema marca o registro da vaga como "estagiário definido". Ao criar um novo Termo de Compromisso de Estágio (TCE), o operador simplesmente seleciona a vaga correspondente. O sistema puxa todos os dados cadastrados na vaga (plano de atividades, cargo do supervisor, horário, valor da bolsa e auxílio transporte) e preenche automaticamente o formulário do contrato, tornando os campos de dados contratuais apenas-leitura (readonly) para manter conformidade.

Capítulo 09

Módulo de Avaliações de Desempenho

Em conformidade com a legislação federal brasileira, os estagiários precisam ter seu desempenho avaliado periodicamente pelos supervisores.

Geração e Agendamento Automático

O sistema gera avaliações de desempenho em dois momentos específicos:

  • Periódica (6 Meses): Gerada de forma automatizada quando o termo de estágio ativo completa 6 meses de vigência.
  • Finalização/Rescisão: Gerada manualmente pelo operador no momento em que um termo é rescindido.

A rotina agendada diária GerarAvaliacoesAutomaticasJob executa no servidor Laravel às 02:00, varrendo os contratos, calculando a periodicidade e gerando registros na tabela tb_avaliacoes com as perguntas padrão em formato JSON.

Formulários Sem Login Necessário (Segurança via Tokens)

Para evitar que supervisores externos tenham que realizar cadastro e login na plataforma apenas para responder a avaliação, o sistema gera links públicos temporários.

O link contém um token de segurança de 32 bytes gerado via bin2hex(random_bytes(32)). O supervisor acessa o link, preenche as notas (de 1 a 5) e feedbacks textuais, valida seu e-mail e submete a resposta. No exato instante do salvamento das respostas, o token é permanentemente invalidado e deletado do banco, impedindo novos acessos a essas informações.

Capítulo 10

Cálculo e Processamento de Folha de Pagamento

Salvamento em Lote (Processamento Batch)

Para garantir que o processamento mensal da remuneração de milhares de estagiários não gere estouros de memória PHP ou timeout de banco de dados, o sistema divide os salvamentos das parcelas em lotes. Cada lote processa, calcula as horas/dias úteis e insere até 50 registros por inserção em lote na tabela tb_folhas_termos.

Geração de Remessas Bancárias

Uma vez consolidada a folha mensal de pagamento (Bolsa Auxílio + Auxílio Transporte - Descontos), o operador pode gerar arquivos no layout de remessa bancária nacional. O sistema permite agrupar os lançamentos por localidade de trabalho ou empresa concedente e efetua a divisão automática em múltiplos arquivos de lote para importação direta no internet banking da cooperativa/banco de pagamento.

Capítulo 11

Módulo de Chamados e Suporte Técnico

Para centralizar a comunicação de alterações contratuais e suporte de dúvidas das empresas concedentes, o SIGE implementa um módulo de chamados operacionais.

Tipagem Contextual de Chamados

Quando a empresa abre um chamado, ela define seu escopo:

  • Rescisão de Contrato: Exige preenchimento obrigatório da data final pretendida e motivo específico da rescisão contratual.
  • Alteração de Termo: Exige a escolha do estagiário e descrição das alterações solicitadas (bolsa, horário, atividades).
  • Outros: Chamado geral de livre preenchimento para dúvidas ou suporte de sistema, com suporte ao envio de até 5 arquivos anexos.

Chat com Loading State e Anexos

O painel de detalhes do chamado renderiza um chat bidirecional em tempo real. O botão de envio do chat desabilita-se instantaneamente exibindo um spinner de loading após o clique, garantindo que o usuário não submeta mensagens repetidas em redes instáveis. O chat suporta até 5 anexos de no máximo 5MB cada nos formatos de imagem, PDF ou documentos corporativos.

Notificações Inteligentes por E-mail

Para evitar excesso de e-mails em chamados com responsável definido, o backend gerencia notificações da seguinte forma:

  • Se o chamado tiver um operador responsável associado na tabela tb_chamados, apenas este operador recebe o e-mail de alerta de novas mensagens.
  • Se o chamado não tiver responsável, todos os operadores e administradores cadastrados recebem a notificação de nova mensagem no chat.
Capítulo 12

Módulo SIGE Concursos (Processos Seletivos)

Isolamento e Cadastro

Para garantir que o módulo de concursos públicos não misture dados cadastrais com o sistema de gestão de estágios padrão, o SIGE Concursos trabalha com tabelas próprias utilizando o prefixo sigeconcursos_tb_. O cadastro abrange Órgãos Públicos, Processos (Editais), Cargos vinculados com remuneração e taxas individuais, locais de prova física e salas.

Algoritmo de Distribuição por Salas

O sistema possui rotina automatizada que distribui os candidatos homologados nos assentos disponíveis das salas de provas físicas:

  1. Mapeia os candidatos de acordo com as opções de localidade e cargos.
  2. Calcula a quantidade mínima necessária de salas em cada local de prova que somadas suportem a capacidade total de inscritos.
  3. Preenche as salas ordenando nominalmente os candidatos em ordem alfabética.
  4. Aplica balanceamento proporcional: se uma sala suporta 50 candidatos mas o local possui 80 inscritos divididos em duas salas, o sistema distribui exatamente 40 em cada sala, mantendo a diferença máxima de no máximo 1 assento de variação entre salas.

Hub Operacional de Status

Em vez de forçar o operador administrativo a alterar manualmente a fase do concurso em várias telas (o que acarreta riscos de inconsistências entre as páginas públicas de candidatos e painéis internos), o sistema calcula a etapa dominante com base nas operações executadas no banco:

  • Inscrições Abertas: Se a data corrente estiver dentro da janela de início e fim.
  • Homologação: Após o fim do período de inscrições e verificação de pagamentos/isenções de taxas.
  • Distribuição de Salas: Quando o operador roda a rotina física de alocação de candidatos.
  • Local de Prova Liberado: Quando a distribuição física é confirmada no backend.

Relatórios de Concursos

Para o dia de aplicação das provas, o módulo gera em PDF as Listas de Presença por Sala, contendo cabeçalho institucional, identificador da sala e listagem nominal dos candidatos com campo para assinatura. A tela de controle de inscritos permite também exportar a listagem de homologações em Excel ou PDF, com opção de censurar os números de CPF dos candidatos para conformidade legal.

Capítulo 13

Políticas de Segurança e Integridade dos Dados

A segurança lógica e a integridade jurídica das transações do sistema são mantidas por meio de:

  • Middleware de Controle de Níveis (RBAC): Bloqueio físico em nível HTTP que impede usuários estagiários e empresas concedentes de acessarem caminhos administrativos (prefixados com /admin ou /operador), independentemente de manipulação de parâmetros.
  • Transações Atômicas de Banco de Dados: Gravações de contratos, pagamentos, chamados e distribuições de salas operam acopladas a transações SQL. Em caso de quedas de conexão do servidor no meio de uma rotina, a transação reverte totalmente (rollback) os dados para o último estado seguro, prevenindo corrupção no banco de dados.
  • Auditoria Jurídica Física: A retenção física e imutável de registros contráteis de estágios em colunas _fixo inviabiliza alterações fraudulentas retroativas e assegura a integridade das auditorias governamentais.
Capítulo 14

Manual de Instalação e Execução Técnica (Setup)

Requisitos do Servidor Operacional

  • Servidor Apache com módulo mod_rewrite ativo, ou servidor Nginx.
  • PHP 8.1 ou superior com extensões: pdo_mysql, mbstring, xml, gd, zip, openssl.
  • Banco de Dados MySQL 5.7+ ou MariaDB 10.3+.
  • Composer e Node.js/NPM instalados no servidor para build.

Comandos de Inicialização e Provisionamento

Para provisionar o sistema de forma limpa em ambiente de produção, execute a sequência de comandos descrita abaixo: 

# Instalar as dependências do backend PHP de forma otimizada
composer install --no-dev --optimize-autoloader

# Instalar dependências e compilar assets de frontend (Tailwind/Vite)
npm install && npm run build

# Executar as migrações estruturais das tabelas de banco de dados
php artisan migrate

# Popular tabelas auxiliares essenciais (Cidades, Estados, Tipos de Chamados)
php artisan db:seed

# Criar link simbólico para a pasta pública de storage
php artisan storage:link
